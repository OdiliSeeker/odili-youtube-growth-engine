## Architecture decisions

- **Public vs admin split:** only the funnel endpoints listed above are public; everything else requires `x-api-key`
- **Deterministic-first + AI-enriched + quota-resilient** (Social/Viral/Growth/content-plan layers): pure-Python templates always produce usable output; AI only upgrades it; any AI failure silently keeps the deterministic result â€” these endpoints NEVER 402. Scores are pure-Python heuristics
- Nothing auto-posts and nothing bulk-sends without explicit admin approval
- **Video grid (`video_grid_service` + `VideoGridItem`):** 6 fixed categories, deterministic (no AI). URLs guarded via `_safe_url` (`^https?:`). Weekly rotation cycles each category's pool by ISO week. Landing renders it OPEN (engage-first) â€” no email gate; `loadExploreGrid` renders only non-empty categories so there are never empty sections. Grid is seeded via admin `POST /admin/video-grid` (no YouTube API â€” the Replit YouTube connector shares a Google quota pool that is frequently exhausted, so auto-sync is unreliable; seed URLs manually)
- **SEO engine (`seo_service` + `Article`):** deterministic-first + AI-enrich, NEVER 402 â€” AI failure silently keeps the template output. `/seo/article` publishes an Article to `/truth/{slug}`; article `video_url` guarded via `_safe_url`. Public blog pages `html.escape` all stored article fields (XSS)
- HMAC-signed unsubscribe tokens via `SESSION_SECRET` â€” no DB storage. Startup validation warns on bad secrets, never crashes
- All emails share one branded shell (`email_shell`, dark/gold). Logo must be an absolute `https://<domain>/static/logo.png` URL (`_email_urls()`). Preview and send use the same renderer. All email YouTube links carry `?src=email&utm=odili_email` (`_with_email_tracking`)
- Video-announcement emails use `generate_evangelization_email` (curiosity subject + hook + short teaser, no script dump; opening line = video hook when supplied). AI weekly newsletter (`generate_newsletter_content`) is a separate format
- **Schema evolution:** new tables auto-create via `create_all`; new *columns on existing tables* need an explicit ALTER in `_ensure_columns` (`db.init_db`, PRAGMA-checked) â€” covers Subscriber `interest`/`drip_step`, Topic `sort_order`
- **5-email drip (`drip_service.py`), offsets `[0,1,3,5,7]` days:** state = `Subscriber.drip_step`, due = `subscribed_at + offset[step]`. Two senders (`start_drip` on subscribe + 30-min scheduler `process_due_drips`) both claim steps with an atomic compare-and-set UPDATE so email 1 can't double-send; failures release the step for retry. **Drip/queue/scheduler only fire while the server runs â†’ needs an always-on (Reserved VM) deployment.** CSV import is drip-free. - Signup tagging (`tag_service`, unique `(subscriber_id, tag)`): every signup â†’ `new-lead`; source + interest tags. `source` validated in `SubscribeRequest`
